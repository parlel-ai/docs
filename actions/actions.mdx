## Overview

Actions are what separate Parlel from a knowledge-only chatbot. An action is a real transaction — a subscription change, a contact detail update, a support request — executed against your backend systems in real time. Customers don't get told to "visit the portal" or "call support." They complete the task in the AI tool they're already using.

Every action is verified before execution, logged with a full audit trail, and executed only when the customer's authentication tier meets the requirement.

## How it works

Actions are defined as configurable flows in the Parlel portal. Each flow specifies what data to collect, what validation to apply, what API call to make, and how to handle success and error states.

When a customer requests an action in conversation, the AI confirms the request before Parlel executes it. Execution only happens after explicit confirmation. The result is logged as a [Transaction](/transactions/transactions).

## Action classes

Actions have two classes that determine when they can fire:

| Class | When it fires |
|---|---|
| General | Any point in an authenticated conversation |
| Journey | Only within an active Journey stage that explicitly permits it |

General actions are always available once the customer has the required authentication tier. Journey actions are gated by journey configuration — they only become available when the customer has reached the right stage.

## Authentication requirements

Every action has a minimum authentication tier. Parlel enforces this automatically and prompts the customer to verify if they haven't reached the required tier.

| Tier | Description |
|---|---|
| Tier 1 | Customer identified — email looked up in CRM |
| Tier 2 | Customer OTP-verified — identity confirmed via code |

Sensitive actions — anything that changes customer data or has financial implications — should require Tier 2. Read-only lookups can operate at Tier 1.

See [Authentication](/authentication/authentication) for the full tier model.

## Configuration

### Creating an action

Go to **Actions** in the portal → **New action**. Define:

1. **Name and description** — how the AI understands what this action does
2. **Action class** — general or journey
3. **Required authentication tier** — minimum tier before execution
4. **Flow** — the sequence of steps: data collection, validation, API call, response handling

### Building a flow

Each action has a flow defined in the visual flow editor. A flow consists of nodes:

- **Input nodes** — collect data from the customer or session
- **API nodes** — make calls to connected integrations
- **Condition nodes** — branch based on API responses
- **Response nodes** — define what the AI communicates back

### Testing an action

Use the test interface in the portal to run an action flow against your integration in a controlled environment before making it available to customers.
